# Go 进阶篇 03 — 包与模块（Package & Module）

> **适用版本**：Go 1.24 | **面向读者**：有编程经验的 SRE / DevOps 工程师

## 导读

Go 的包（Package）和模块（Module）系统是组织代码的基石。对于 SRE/DevOps 工程师来说，理解包和模块管理至关重要：
- 编写可复用的运维工具库
- 管理复杂项目的依赖关系
- 处理私有仓库的依赖（企业内部 Go 库）
- 使用工作区（workspace）同时开发多个关联项目

本篇将从 package 声明和导入规则开始，深入 go mod 命令、go.mod/go.sum 文件、语义版本控制、依赖替换、工作区等核心概念。

## 1. Package 基础

### 1.1 包声明与导入

```go
// 文件：main.go
package main // 每个 Go 文件必须以 package 声明开头

// 导入单个包
// import "fmt"

// 导入多个包（推荐使用分组导入）
import (
	"fmt"
	"math"
	"strings"
)

func main() {
	// 使用包名.函数名的方式调用
	fmt.Println("Hello, SRE!")
	fmt.Println("圆周率:", math.Pi)
	fmt.Println(strings.ToUpper("devops"))
}
```

### 1.2 包的命名规则

```go
package main

import "fmt"

/*
Go 包命名规则和最佳实践：

1. 包名应该简短、小写、单个单词
   好的: net, http, os, io, fmt, sync
   不好的: stringUtilities, my_package, httpHelpers

2. 包名应该与目录名一致
   目录 /utils/config/ → package config
   目录 /internal/auth/ → package auth

3. 包名不要与标准库冲突
   不要: 你的项目里叫 package fmt
   不要: 你的项目里叫 package http

4. 包名是类型和函数的命名空间
   好的: config.Load()（而不是 config.LoadConfig()）
   好的: http.Client（而不是 http.HTTPClient）

5. 避免 utils、common、base 等无意义包名
   不好的: utils.FormatDate()
   好的: timeutil.Format() 或直接放在使用它的包里

6. 测试文件的包名
   普通测试: package mypackage（可以测试未导出的函数）
   黑盒测试: package mypackage_test（只能测试导出的 API）
*/

func main() {
	fmt.Println("Go 包命名规则示例")
	fmt.Println("具体规则请参阅本文件中的注释")
}
```

### 1.3 包的导入路径

```go
package main

import (
	"fmt"

	// 标准库导入
	"encoding/json"
	"net/http"
	"os/exec"

	// 第三方库导入（使用完整的模块路径）
	// "github.com/spf13/cobra"
	// "github.com/prometheus/client_golang/prometheus"

	// 项目内部导入
	// "mycompany.com/myproject/internal/config"
	// "mycompany.com/myproject/pkg/logger"
)

// 导入别名
// import (
//     stdhttp "net/http"                    // 给标准库起别名
//     k8s "k8s.io/client-go/kubernetes"     // 简化长路径
//     _ "github.com/lib/pq"                 // 空白导入：只执行 init()
//     . "math"                              // 点导入：可以直接用 Pi 而不是 math.Pi（不推荐）
// )

func main() {
	// 标准库使用
	fmt.Println("标准库包:")
	data, _ := json.Marshal(map[string]string{"role": "sre"})
	fmt.Println("  json:", string(data))
	fmt.Println("  http 默认客户端:", http.DefaultClient)

	out, _ := exec.Command("echo", "hello").Output()
	fmt.Printf("  exec: %s", out)
}
```

## 2. 可见性（大小写控制导出）

### 2.1 导出 vs 未导出

```go
package main

import "fmt"

// Go 使用首字母大小写来控制可见性，这是 Go 最独特的设计之一

// ========== 模拟一个 "server" 包 ==========

// Server 首字母大写 → 导出（public），包外可访问
type Server struct {
	Name string // 导出字段
	Host string // 导出字段
	Port int    // 导出字段

	started bool    // 首字母小写 → 未导出（private），包外不可访问
	connCount int   // 未导出字段
}

// NewServer 导出的构造函数
func NewServer(name, host string, port int) *Server {
	return &Server{
		Name: name,
		Host: host,
		Port: port,
	}
}

// Start 导出的方法
func (s *Server) Start() error {
	s.started = true
	fmt.Printf("服务器 %s 启动在 %s:%d\n", s.Name, s.Host, s.Port)
	return nil
}

// IsRunning 导出的方法
func (s *Server) IsRunning() bool {
	return s.started
}

// addConnection 未导出的方法（只能在同一个包内使用）
func (s *Server) addConnection() {
	s.connCount++
}

// formatAddr 未导出的辅助函数
func formatAddr(host string, port int) string {
	return fmt.Sprintf("%s:%d", host, port)
}

func main() {
	srv := NewServer("api-gateway", "0.0.0.0", 8080)
	srv.Start()
	fmt.Println("运行中:", srv.IsRunning())

	// 在同一个包内，可以访问未导出的成员
	srv.addConnection()
	fmt.Println("连接数:", srv.connCount)
	fmt.Println("地址:", formatAddr(srv.Host, srv.Port))
}
```

### 2.2 可见性最佳实践

```go
package main

import "fmt"

/*
可见性最佳实践：

1. 尽量少导出
   - 导出的东西成为 API 的一部分，变更需要考虑向后兼容
   - 未导出的可以随时重构

2. 导出类型，但限制字段导出
   好的做法：使用构造函数 + getter/setter
*/

// Config 导出类型
type Config struct {
	Host     string // 导出：允许直接访问
	Port     int    // 导出：允许直接访问
	password string // 未导出：敏感信息不暴露
	maxRetry int    // 未导出：内部实现细节
}

// NewConfig 导出的构造函数——提供合理的默认值
func NewConfig(host string, port int) *Config {
	return &Config{
		Host:     host,
		Port:     port,
		password: "",
		maxRetry: 3,
	}
}

// SetPassword 通过方法设置敏感字段
func (c *Config) SetPassword(pwd string) {
	c.password = pwd
}

// MaxRetry 通过方法获取内部字段
func (c *Config) MaxRetry() int {
	return c.maxRetry
}

// SetMaxRetry 通过方法设置内部字段（可以加入校验）
func (c *Config) SetMaxRetry(n int) error {
	if n < 1 || n > 10 {
		return fmt.Errorf("maxRetry 必须在 1-10 之间, 当前: %d", n)
	}
	c.maxRetry = n
	return nil
}

func main() {
	cfg := NewConfig("localhost", 5432)
	cfg.SetPassword("secret123")

	if err := cfg.SetMaxRetry(5); err != nil {
		fmt.Println("错误:", err)
	}

	fmt.Printf("Host: %s, Port: %d, MaxRetry: %d\n",
		cfg.Host, cfg.Port, cfg.MaxRetry())

	// 错误的重试次数
	if err := cfg.SetMaxRetry(100); err != nil {
		fmt.Println("错误:", err) // maxRetry 必须在 1-10 之间, 当前: 100
	}
}
```

## 3. Go Modules 命令详解

### 3.1 go mod init — 初始化模块

```bash
# 创建新项目
mkdir myproject && cd myproject

# 初始化模块（模块路径通常是代码仓库路径）
go mod init github.com/mycompany/myproject

# 对于不发布的内部工具，可以使用简单路径
go mod init mycompany.com/internal-tool

# 会生成 go.mod 文件
```

### 3.2 go mod tidy — 整理依赖

```bash
# go mod tidy 做两件事：
# 1. 添加代码中引用但 go.mod 中缺失的依赖
# 2. 删除 go.mod 中有但代码不再引用的依赖

go mod tidy

# 常见使用场景：
# - 添加了新的 import 后运行
# - 删除了一些代码后运行
# - 从别人那里拉取代码后运行
# - CI/CD 流水线中运行（检查是否干净）

# 在 CI 中检查 go.mod 是否已整理
# go mod tidy && git diff --exit-code go.mod go.sum
```

### 3.3 go mod vendor — 依赖本地化

```bash
# 将所有依赖复制到项目的 vendor/ 目录
go mod vendor

# 使用 vendor 目录构建（Go 1.14+ 如果存在 vendor/ 会自动使用）
go build -mod=vendor ./...

# vendor 的使用场景：
# - 离线构建环境
# - 需要审计依赖代码
# - 镜像构建时减少网络依赖
# - 企业安全合规要求
```

### 3.4 其他 go mod 命令

```bash
# 下载依赖到本地缓存（$GOPATH/pkg/mod/cache/）
go mod download

# 查看依赖关系图
go mod graph

# 查看某个依赖被谁引用
go mod why github.com/some/package

# 验证依赖完整性
go mod verify

# 编辑 go.mod 文件（程序化方式）
go mod edit -require github.com/pkg/errors@v0.9.1
go mod edit -replace github.com/old/pkg=github.com/new/pkg@v1.0.0
go mod edit -droprequire github.com/unused/pkg
```

### 3.5 完整示例：创建和管理一个项目

```go

/*
完整的项目创建流程：

# 1. 创建项目目录
```

## 4. go.mod 文件详解

### 4.1 go.mod 结构

```go
package main

import "fmt"

/*
go.mod 文件示例和各部分说明：

------- go.mod 开始 -------

// 模块声明：这个项目的导入路径
module github.com/mycompany/deploy-tool

// Go 版本要求
go 1.24

// 直接依赖（你的代码直接 import 的包）
require (
    github.com/spf13/cobra v1.8.0
    github.com/prometheus/client_golang v1.19.0
    k8s.io/client-go v0.29.2
    gopkg.in/yaml.v3 v3.0.1
)

// 间接依赖（你的直接依赖所依赖的包）
require (
    github.com/inconshreveable/mousetrap v1.1.0 // indirect
    github.com/spf13/pflag v1.0.5 // indirect
    golang.org/x/net v0.21.0 // indirect
    golang.org/x/sys v0.17.0 // indirect
)

// 依赖替换（见后文详解）
replace (
    github.com/old/package => github.com/new/package v1.0.0
)

// 排除特定版本
exclude (
    github.com/buggy/pkg v1.2.3
)

// 依赖回退（Go 1.17+，指定某个依赖的最低版本）
retract (
    v1.0.0 // 这个版本有严重 bug
    [v1.1.0, v1.2.0] // 这个范围的版本都有问题
)

------- go.mod 结束 -------
*/

func main() {
	fmt.Println("go.mod 文件结构说明")
	fmt.Println()

	sections := []struct {
		name, desc string
	}{
		{"module", "声明当前模块的导入路径"},
		{"go", "指定最低 Go 版本"},
		{"require", "列出所有依赖及其版本"},
		{"require (indirect)", "间接依赖，由 go mod tidy 自动管理"},
		{"replace", "替换依赖（本地开发、fork 等）"},
		{"exclude", "排除特定的依赖版本"},
		{"retract", "撤回已发布的版本"},
	}

	for _, s := range sections {
		fmt.Printf("  %-25s %s\n", s.name, s.desc)
	}
}
```

### 4.2 go.sum 文件

```go
package main

import "fmt"

/*
go.sum 文件说明：

go.sum 是依赖的校验和文件，确保依赖不被篡改。

格式：
  模块路径 版本 哈希算法:哈希值
  模块路径 版本/go.mod 哈希算法:哈希值

示例：
  github.com/spf13/cobra v1.8.0 h1:xxxx...
  github.com/spf13/cobra v1.8.0/go.mod h1:yyyy...

关键要点：
1. go.sum 应该提交到 Git（它是安全保证）
2. 不要手动编辑 go.sum
3. go mod tidy 会自动更新 go.sum
4. go mod verify 可以验证本地缓存是否与 go.sum 一致
5. go.sum 可能包含的条目比 go.mod 多（包含历史依赖的校验和）
*/

func main() {
	fmt.Println("go.sum 文件是依赖完整性的保障")
	fmt.Println()
	fmt.Println("常见操作:")
	fmt.Println("  go mod tidy     # 自动更新 go.sum")
	fmt.Println("  go mod verify   # 验证依赖完整性")
	fmt.Println("  go mod download # 下载并缓存依赖")
	fmt.Println()
	fmt.Println("CI/CD 中检查依赖是否干净:")
	fmt.Println("  go mod tidy && git diff --exit-code go.mod go.sum")
}
```

## 5. 语义版本控制（Semver）

### 5.1 版本号规则

```go
package main

import "fmt"

func main() {
	fmt.Println("Go Modules 使用语义版本控制 (Semantic Versioning)")
	fmt.Println()

	/*
	版本格式: vMAJOR.MINOR.PATCH

	v1.2.3
	│ │ │
	│ │ └── PATCH: 向后兼容的 Bug 修复
	│ └──── MINOR: 向后兼容的新功能
	└────── MAJOR: 不兼容的 API 变更
	*/

	versions := []struct {
		version, meaning string
	}{
		{"v0.1.0", "初始开发版本，API 不稳定，随时可能变更"},
		{"v1.0.0", "首个稳定版本，承诺 API 兼容性"},
		{"v1.1.0", "新增功能，向后兼容"},
		{"v1.1.1", "Bug 修复，向后兼容"},
		{"v2.0.0", "不兼容的 API 变更（导入路径需要加 /v2）"},
	}

	for _, v := range versions {
		fmt.Printf("  %-10s %s\n", v.version, v.meaning)
	}

	fmt.Println()
	fmt.Println("Go 的特殊规则:")
	fmt.Println("  v0.x.x 和 v1.x.x: 导入路径不需要版本号后缀")
	fmt.Println("    import \"github.com/pkg/errors\"")
	fmt.Println()
	fmt.Println("  v2.x.x 及以上: 导入路径必须带版本号后缀")
	fmt.Println("    import \"github.com/pkg/errors/v2\"")
	fmt.Println("    go.mod 中: module github.com/pkg/errors/v2")
	fmt.Println()
	fmt.Println("版本选择:")
	fmt.Println("  go get github.com/pkg/errors@v0.9.1     # 指定版本")
	fmt.Println("  go get github.com/pkg/errors@latest      # 最新稳定版")
	fmt.Println("  go get github.com/pkg/errors@main        # 指定分支")
	fmt.Println("  go get github.com/pkg/errors@abc1234     # 指定提交")
}
```

### 5.2 版本选择与最小版本选择（MVS）

```go
package main

import "fmt"

func main() {
	fmt.Println("Go 的最小版本选择 (Minimal Version Selection, MVS)")
	fmt.Println()

	/*
	与 npm/pip 不同，Go 使用 MVS 算法：

	假设你的项目依赖：
	  A requires B >= v1.2.0
	  C requires B >= v1.3.0

	MVS 选择: B v1.3.0（满足所有需求的最小版本）
	npm 可能选择: B v1.5.0（最新版本）

	MVS 的优势：
	1. 构建可重复——今天和明天构建的结果一样
	2. 不会引入不必要的新版本
	3. 减少了意外破坏的风险
	*/

	fmt.Println("MVS 示例:")
	fmt.Println()
	fmt.Println("你的 go.mod:")
	fmt.Println("  require github.com/A v1.0.0")
	fmt.Println("  require github.com/B v1.0.0")
	fmt.Println()
	fmt.Println("A 的 go.mod:")
	fmt.Println("  require github.com/C v1.2.0")
	fmt.Println()
	fmt.Println("B 的 go.mod:")
	fmt.Println("  require github.com/C v1.5.0")
	fmt.Println()
	fmt.Println("MVS 最终选择 C v1.5.0（满足 A 和 B 需求的最小版本）")
	fmt.Println()
	fmt.Println("对比其他包管理器：")
	fmt.Println("  npm: 可能选择 C v1.9.0（最新的 ^1.x.x）")
	fmt.Println("  pip: 可能选择 C v1.9.0（满足 >=1.2.0 的最新版本）")
	fmt.Println("  Go MVS: 一定选择 C v1.5.0（不多不少）")
}
```

## 6. 依赖替换 replace

### 6.1 replace 的使用场景

```go
package main

import "fmt"

func main() {
	fmt.Println("go.mod replace 指令的使用场景")
	fmt.Println()

	scenarios := []struct {
		scenario, example, desc string
	}{
		{
			"本地开发/调试",
			"replace github.com/mycompany/shared-lib => ../shared-lib",
			"开发时使用本地路径，而非远程仓库",
		},
		{
			"Fork 替换",
			"replace github.com/original/pkg => github.com/myfork/pkg v1.0.0",
			"使用 fork 版本替换原始包",
		},
		{
			"修复上游 Bug",
			"replace github.com/buggy/pkg => github.com/myfork/pkg v0.0.0-20240101-abcdef",
			"上游还没合并你的 PR 时使用",
		},
		{
			"版本降级",
			"replace github.com/pkg/v2 v2.1.0 => github.com/pkg/v2 v2.0.0",
			"强制使用旧版本（新版本有 bug）",
		},
	}

	for i, s := range scenarios {
		fmt.Printf("%d. %s\n", i+1, s.scenario)
		fmt.Printf("   go.mod: %s\n", s.example)
		fmt.Printf("   说明: %s\n\n", s.desc)
	}
}
```

### 6.2 实际操作示例

```bash
# 场景1: 本地开发
# 假设你同时开发两个项目：deploy-tool 和 shared-lib
# 目录结构：
#   ~/projects/deploy-tool/   (主项目)
#   ~/projects/shared-lib/    (共享库)

# 在 deploy-tool 的 go.mod 中添加：
cd ~/projects/deploy-tool
go mod edit -replace github.com/mycompany/shared-lib=../shared-lib

# 场景2: 使用 fork
go mod edit -replace github.com/original/pkg=github.com/myfork/pkg@v1.0.0

# 场景3: 发布前移除 replace（重要！）
# replace 中使用本地路径的模块无法被其他人使用
go mod edit -dropreplace github.com/mycompany/shared-lib
```

## 7. 工作区 go work（Go 1.18+）

### 7.1 工作区基础

```go
package main

import "fmt"

func main() {
	fmt.Println("Go 工作区 (Workspace) — Go 1.18+ 引入")
	fmt.Println()

	/*
	工作区解决了什么问题？
	在开发多个相互依赖的模块时，以前需要在每个模块的 go.mod 中使用 replace
	指令指向本地路径。发布前还要记得删掉这些 replace。

	工作区提供了更优雅的方案：在工作区级别统一管理多个模块的关系，
	不需要修改各个模块的 go.mod。
	*/

	fmt.Println("创建工作区:")
	fmt.Println("  mkdir workspace && cd workspace")
	fmt.Println("  go work init ./module-a ./module-b")
	fmt.Println()
	fmt.Println("这会创建 go.work 文件：")
	fmt.Println("  go 1.24")
	fmt.Println("  use (")
	fmt.Println("      ./module-a")
	fmt.Println("      ./module-b")
	fmt.Println("  )")
	fmt.Println()
	fmt.Println("常用命令:")
	fmt.Println("  go work init [modules...]  # 初始化工作区")
	fmt.Println("  go work use [modules...]   # 添加模块到工作区")
	fmt.Println("  go work sync               # 同步依赖版本")
	fmt.Println("  go work edit               # 编辑 go.work 文件")
}
```

### 7.2 工作区实战

```bash
# 目录结构：
```

## 9. internal 包

### 9.1 internal 包的访问限制

```go
package main

import "fmt"

/*
Go 的 internal 包提供了比"未导出"更强的访问控制。

规则：
internal 包中的代码只能被 internal 目录的父目录及其子目录导入。

目录结构示例：
myproject/
├── go.mod
├── cmd/
│   └── server/
│       └── main.go         # 可以导入 internal/
├── internal/                # 项目级 internal
│   ├── config/
│   │   └── config.go       # 只有 myproject 内的代码能用
│   └── database/
│       └── db.go           # 只有 myproject 内的代码能用
├── pkg/                     # 公开的包（任何人可用）
│   └── api/
│       ├── handler.go
│       └── internal/        # api 包级别的 internal
│           └── helper.go   # 只有 pkg/api/ 内的代码能用
└── service/
    ├── user.go              # 可以导入 internal/config
    └── internal/            # service 包级别的 internal
        └── cache.go         # 只有 service/ 内的代码能用
*/

func main() {
	fmt.Println("internal 包的访问控制规则")
	fmt.Println()

	rules := []struct {
		importer, target, allowed string
	}{
		{"cmd/server/main.go", "internal/config", "允许（internal 的父目录 myproject 的子目录）"},
		{"service/user.go", "internal/config", "允许（同上）"},
		{"pkg/api/handler.go", "internal/config", "允许（同上）"},
		{"pkg/api/handler.go", "pkg/api/internal/helper", "允许（handler 在 internal 的父目录下）"},
		{"service/user.go", "pkg/api/internal/helper", "禁止（service 不在 pkg/api 下）"},
		{"外部项目", "internal/config", "禁止（外部项目不在 myproject 下）"},
		{"外部项目", "pkg/api/handler.go", "允许（pkg 是公开的）"},
	}

	for _, r := range rules {
		fmt.Printf("  %-30s → %-30s %s\n", r.importer, r.target, r.allowed)
	}
}
```

### 10.2 避免循环依赖

```go
package main

import "fmt"

/*
Go 不允许循环依赖（编译错误）。

问题示例：
  package user imports package order
  package order imports package user
  → 编译错误: import cycle not allowed

解决方案：

1. 接口提取法
   创建一个共享的接口包，两个包都依赖接口而不是具体实现

   types/      → 定义 UserFinder 接口和 OrderFinder 接口
   user/       → 实现 UserFinder，依赖 types.OrderFinder
   order/      → 实现 OrderFinder，依赖 types.UserFinder

2. 依赖注入法
   通过函数参数或结构体字段注入依赖

3. 合并包
   如果两个包强耦合，考虑合并为一个包

4. 事件/回调
   使用事件模式解耦
*/

// 演示依赖注入法解决循环依赖

// 模拟 user 包
type UserService struct {
	findOrdersFn func(userID int) []string
}

func NewUserService(findOrders func(userID int) []string) *UserService {
	return &UserService{findOrdersFn: findOrders}
}

func (s *UserService) GetUserWithOrders(userID int) {
	fmt.Printf("用户 %d 的订单: %v\n", userID, s.findOrdersFn(userID))
}

// 模拟 order 包
type OrderService struct{}

func (s *OrderService) FindByUser(userID int) []string {
	return []string{"ORD-001", "ORD-002"}
}

func main() {
	orderSvc := &OrderService{}

	// 通过函数注入解决循环依赖
	userSvc := NewUserService(orderSvc.FindByUser)
	userSvc.GetUserWithOrders(42)
}
```

### 10.3 init() 函数

```go
package main

import "fmt"

/*
init() 函数的规则：
1. 每个包可以有多个 init() 函数
2. init() 在包被导入时自动执行
3. init() 在所有包级变量初始化之后执行
4. init() 不能被显式调用
5. 执行顺序：依赖包的 init() 先于当前包的 init()

执行顺序：
  导入的包的全局变量初始化
  → 导入的包的 init()
  → 当前包的全局变量初始化
  → 当前包的 init()
  → main()
*/

var startTime string

func init() {
	// init 在 main 之前执行
	startTime = "初始化完成"
	fmt.Println("init() #1: 初始化全局状态")
}

func init() {
	// 同一个文件可以有多个 init
	fmt.Println("init() #2: 注册默认配置")
}

func main() {
	fmt.Println("main(): 程序开始")
	fmt.Println("startTime:", startTime)

	fmt.Println()
	fmt.Println("init() 的常见用途:")
	fmt.Println("  1. 注册数据库驱动: import _ \"github.com/lib/pq\"")
	fmt.Println("  2. 注册图片解码器: import _ \"image/png\"")
	fmt.Println("  3. 初始化全局变量（如编译好的正则表达式）")
	fmt.Println()
	fmt.Println("init() 的注意事项:")
	fmt.Println("  1. 不要在 init 中做耗时操作")
	fmt.Println("  2. 不要在 init 中依赖其他包的 init 顺序")
	fmt.Println("  3. 优先使用显式初始化函数而非 init")
	fmt.Println("  4. 空白导入(import _)只在注册驱动等场景使用")
}
```

## 常见坑点总结

| 坑点 | 说明 | 解决方案 |
|------|------|----------|
| 循环导入 | 包 A 依赖 B，B 又依赖 A | 提取接口到独立包，或合并包 |
| replace 遗留 | 发布时忘记删除本地 replace | 使用 go.work 替代 replace；CI 检查 |
| vendor 过时 | vendor 目录与 go.mod 不同步 | CI 中运行 `go mod vendor && git diff` |
| 版本不兼容 | v2+ 的包忘记修改导入路径 | v2+ 必须在导入路径和 module 中加 `/v2` |
| init() 副作用 | init 中做了太多事 | 尽量使用显式初始化函数 |
| GOPRIVATE 未设置 | 拉取私有包失败 | 设置 GOPRIVATE 环境变量 |
| 包名与目录名不一致 | 容易混淆 | 保持包名和目录名一致 |

## 速查表 / 小结

### go mod 命令速查

| 命令 | 用途 |
|------|------|
| `go mod init MODULE` | 初始化模块 |
| `go mod tidy` | 整理依赖（添加缺失、删除多余） |
| `go mod vendor` | 将依赖复制到 vendor/ |
| `go mod download` | 下载依赖到本地缓存 |
| `go mod graph` | 查看依赖关系图 |
| `go mod verify` | 验证依赖完整性 |
| `go mod why PKG` | 查看为什么需要某个依赖 |
| `go mod edit -replace` | 添加依赖替换 |
| `go get PKG@VERSION` | 添加/更新依赖 |

### go work 命令速查

| 命令 | 用途 |
|------|------|
| `go work init [MODULES...]` | 创建工作区 |
| `go work use MODULE` | 添加模块到工作区 |
| `go work sync` | 同步依赖版本 |
